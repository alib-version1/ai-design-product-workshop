# Sample Service Blueprint: Checking Application Progress

This is an example service blueprint created using M365 Copilot and refined with team input. The format is a table showing what happens at each stage of the journey, across front-stage, back-stage, and support layers.

---

## Service blueprint

| Layer | Application submitted | Citizen wants an update | Citizen tries to find information | Citizen contacts the service | Citizen gets an answer (or not) | Citizen continues waiting |
|-------|----------------------|------------------------|----------------------------------|-----------------------------|---------------------------------|--------------------------|
| **User actions** | Fills in form, submits, waits for confirmation | Feels anxious, decides to check | Looks for tracking page, searches website, checks email | Phones contact centre, waits on hold | Listens to agent, asks follow-up questions | Checks email, considers calling again |
| **Touchpoints** | Online form, post | None available | Website, search engine | Phone | Phone | Email, phone |
| **Visible staff actions** | None visible to citizen | None | None | Agent answers, looks up case | Agent reads status from system, relays to citizen | None |
| **Back-stage actions** | Application logged in case management system, assigned to queue | Case sits in queue or is being reviewed | No proactive communication triggered | Agent searches case management system | Agent interprets status codes, may escalate to caseworker | Caseworker continues processing when able |
| **Support processes** | Case management system, document storage, postal service | Queue management, workload allocation | Website CMS (no status feature exists) | Telephony system, CRM, case management system | Case management system, internal guidance | Case management system, policy rules, third-party checks |
| **Citizen thinking/feeling** | "I hope that went through OK" | "It's been two weeks, surely someone has looked at it" | "There's no way to check online? Really?" | "40 minutes on hold for this?" | "That's not the same thing they told me last time" | "I just want to know if anyone is actually dealing with it" |

## Key pain points

| Pain point | Where it occurs | Impact |
|-----------|----------------|--------|
| **No confirmation after submission** | Application submitted | Citizen unsure it was received; may submit again or phone to check |
| **No online status tracking** | Citizen tries to find information | Forces citizen to phone, creating avoidable contact |
| **Long wait times on the phone** | Citizen contacts the service | Citizen frustration, wasted time, some give up and call back later |
| **Inconsistent information from agents** | Citizen gets an answer | Erodes trust — citizen does not know what to believe |
| **No proactive communication** | Citizen continues waiting | Citizen left guessing; anxiety builds; calls again |

## Where avoidable contact occurs

The majority of avoidable contact happens at the "Citizen contacts the service" stage. If citizens could see their application status online, most of these calls would not happen. A secondary source of avoidable contact is at submission — if citizens received clear confirmation and a reference number, they would not need to phone to check their application was received.
